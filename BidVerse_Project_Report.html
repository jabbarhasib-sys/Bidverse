1. Abstract

 

The world of commerce has changed dramatically in the digital age, and one of the most exciting evolutions has been the move from physical auction houses to fully interactive online auction platforms. BidVerse is a full-stack, real-time web application designed to recreate the thrill and transparency of live bidding — right from the comfort of a browser.

At its core, BidVerse allows registered users to list items for auction, browse active listings, and place competitive bids that are reflected instantly across all connected clients without any page refresh. The platform leverages Socket.IO for bi-directional real-time communication, which ensures that when someone places a bid, every other participant watching the same auction sees the updated price within milliseconds.

What makes BidVerse stand out from a simple CRUD application is its live notification system. When an auction creator decides to close bidding and accept the highest offer, the winning bidder receives an immediate on-screen winner announcement — a full-screen golden congratulatory banner — along with a toast popup. This simulates the excitement of a real auction gavel dropping.

The project was built with a modern dark-mode aesthetic using Tailwind CSS and glassmorphism effects, making it visually premium and fully responsive across all screen sizes. The backend is powered by Node.js and Express.js, with data persisted in structured JSON files to eliminate the dependency on external database services during development and testing.

Throughout this report, we walk through every aspect of the system — from the reasoning behind the architectural decisions, to the user-facing design choices, the backend API structure, real-time communication patterns, and the challenges overcome during development.

 

2. Introduction & Features

 

2.1 What is BidVerse?

BidVerse was born from a simple question: What would it feel like to build an auction house on the internet that actually feels alive? Traditional e-commerce platforms like eBay show you a timer and a price, but the experience is static. You place a bid, refresh the page, and hope nobody outbid you in the five seconds it took to load. That experience felt stale.

BidVerse takes a different approach. It treats every browser connected to an auction as a participant in a live event. The moment any user places a bid, all other viewers of that auction see the price update in real time, the current leader changes, and the countdown clock ticks with urgency. The platform rewards attentiveness — the most engaged bidder wins.

The application targets three types of users:

  • Sellers (Auction Creators): People who want to list items and accept bids. They have full control over their auctions and can manually close bidding at any moment to accept the current highest offer.
  • Buyers (Bidders): Users who browse listings, follow auctions they are interested in, and compete with others by placing bids.
  • Guests: Unregistered visitors can browse all active and ended auctions and see bid histories, but must register to participate.

 

2.2 Core Features

 

🔐 User Authentication System

BidVerse implements a complete authentication flow with user registration and login. Passwords are never stored in plain text — they are hashed using bcrypt (10 rounds of salting) before being saved to the database. After login, the server issues a JSON Web Token (JWT) with a 7-day expiry, which the frontend stores in localStorage and attaches to every subsequent API request via the Authorization: Bearer header. This keeps sessions persistent across page refreshes without requiring a server-side session store.

⚡ Real-Time Bidding via Socket.IO

The heart of the platform is its real-time engine. When a user opens an auction detail page, their browser automatically establishes a persistent WebSocket connection to the server using Socket.IO and joins a room named after the auction’s unique ID. When any user submits a bid, the server validates it, saves it to the database, and then broadcasts a new-bid event to every socket in that auction’s room. Every connected browser updates the price, the leader name, and the bid history list — all without a single page reload.

🏆 Winner Notification System

One of the most exciting features of BidVerse is the winner announcement. When a seller clicks “Close Auction & Accept Bid,” the server not only marks the auction as ended but also sends a private auction-won Socket.IO event directly to the winning bidder’s browser. This is achieved by having each logged-in user join a personal socket room (named after their user ID) immediately upon connecting. The winner then sees a full-screen, golden-glowing modal saying “Congratulations! You won the auction for [Item Name]” — making the experience genuinely exciting.

🔍 Live Search & Filtering

The homepage features a real-time search bar that filters auctions as you type, matching against both product names and descriptions. A tab-based filter bar lets users toggle between “All,” “Live,” and “Ended” auctions without any API calls — the filtering is done entirely on the client side using memoized computation for maximum performance.

⏱ Live Countdown Timer

Every active auction displays a live countdown clock that ticks second by second directly in the browser. When an auction enters its final hour, the timer digits turn red and pulse with an urgency message: “⚡ Ending soon — place your bid now!” This creates genuine time pressure that encourages engagement.

📊 Statistics Dashboard

The homepage features a real-time statistics bar showing the number of live auctions, completed auctions, and the total value of all bids placed across the platform. These numbers update every time the page loads, giving users a sense of the platform’s scale and activity.

🖼 Skeleton Loading States

Rather than showing a blank screen while data loads, BidVerse displays animated skeleton placeholder cards that match the shape of the actual auction cards. This makes the interface feel fast and polished, even over slower network connections.

📱 Fully Responsive Design

Every page adapts seamlessly across mobile phones, tablets, and desktop monitors. The auction grid switches between one, two, and three columns depending on screen width, and the bidding interface stacks vertically on small screens.

 

3. Frontend Design

 

3.1 Technology Choices

The frontend of BidVerse is built with React 18 using TypeScript for type safety, bundled by Vite for near-instant hot module replacement during development. The styling is handled by Tailwind CSS compiled through PostCSS, with a heavily customised global stylesheet (index.css) that defines reusable design tokens, animation utilities, and custom component classes.

Google Fonts are used for typography — specifically Inter for body text and Syne for display headings — loaded through the HTML <head> to keep everything self-contained.

 

3.2 Design System & Aesthetic

The entire application is built around a deep dark mode colour palette inspired by deep space and neon glow. The primary background is #070b14 — nearly black but with a subtle blue tint. Content surfaces use #1e293b (slate-800) with semi-transparent “glassmorphism” panels layered on top, created with CSS backdrop-filter: blur() and translucent borders.

The accent colour throughout the platform is cyan (#22d3ee), used for active bid values, buttons, links, and glow effects. Gold (#fbbf24) is reserved exclusively for winner-related elements — the crown icon on the top bidder, the winner banner, and the “🏆 You Won” toast notification. This deliberate colour language makes it immediately obvious who is leading and who has won.

Animations are subtle but meaningful. Every page transition uses a page-enter fade-in with a slight upward translation. Bid amounts “flash” with a golden pulse when they update. The top bidder in the bid history list gets an animated crown (👑) avatar. Live auction badges pulse with a breathing animation to signal that the event is happening right now.

 

3.3 Page Structure

The application has six main pages, each handling a distinct user journey:

Home Page (/)
This is the discovery hub. At the top, a hero section with animated decorative gradient orbs displays the platform tagline and a live count of active auctions. Below it, three stat cards show platform-wide metrics. The main content area is an auction card grid with integrated search and filter controls. Each auction card shows a product image, the current highest bid in a prominent gradient font, the seller’s name, and a countdown or “Ended” badge.

Auction Detail Page (/auctions/:id)
This is where the real action happens. The left half of a two-column grid shows the product image with a gradient overlay, the item description, seller details, and the start/end times. If the viewing user is the one who created the auction and it is still active, a red “Close Auction & Accept Bid” button appears exclusively for them. The right half contains the NoteEditor component — the live bidding panel — which shows the current price, the countdown timer, the bid input form, quick-bid increment buttons, and the full scrollable bid history.

Login Page (/login)
A minimal, centred card with BidVerse branding, email and password fields, and a gradient submit button. Includes a link to registration and graceful error messaging for invalid credentials.

Register Page (/register)
Identical in layout to the login page, with an additional “Name” field. Password hashing happens server-side; the form does no client-side validation beyond basic HTML5 required attributes.

Create Auction Page (/create)
A rich form page that lets authenticated users list a new item. Fields include product name, description, starting price (with a currency symbol prefix), auction end date/time, and an optional image URL. The form has real-time validation feedback and a disabled submit state while the API request is in flight.

My Wins Page (/wins)
A personal history page showing all auctions the logged-in user has won. Each entry shows the item name, winning bid amount, and the date it ended.

3.5 State Management

Global state is minimal and purpose-built. Two React Contexts handle the cross-cutting concerns:

  • AuthProvider (NotesContext.tsx): Manages the logged-in user object and JWT token. Persists to localStorage so sessions survive page refreshes. Exposes login, register, and logout functions.
  • ToastProvider (ToastContext.tsx): Manages a queue of toast notifications with auto-dismiss timers. Any component in the tree can call useToast().toast(message, type) to trigger a notification.

All other state is local to the component that needs it, keeping the application simple and easy to reason about.

 

4. Backend Design

 

4.1 Technology Stack

The backend is a Node.js application using Express.js as the HTTP framework and Socket.IO for real-time WebSocket communication. It is written entirely in ES Module syntax (import/export) and run with nodemon during development for automatic restarts on file changes.

The application is structured as a monorepo, with the backend code living in /backend/src/ and the shared database layer in /database/. This separation was a deliberate architectural choice — the database access code is completely independent of the HTTP layer, making it theoretically possible to swap out the persistence mechanism (e.g., from JSON files to MongoDB) without touching any route or controller logic.

 

4.2 The JSON Database Layer

One of the more unusual architectural decisions in BidVerse is its use of plain JSON files as the persistent data store, rather than a traditional database like MongoDB or PostgreSQL.

This choice was driven by a practical development goal: zero external dependencies. The application starts and works immediately on any machine with Node.js installed, with no need to install MongoDB, configure connection strings, or spin up Docker containers. For a project of this scope — a course assignment with real-time features — this is a significant productivity advantage.

The database layer lives in database/jsonDb.js and exports three “model” objects — User, Auction, and Bid — each with an API that mirrors Mongoose’s query interface. Internally, each model reads the corresponding JSON file from disk (users.json, auctions.json, bids.json), performs the operation in memory, and writes the updated array back to disk atomically.

The file path is resolved using import.meta.url and path.dirname to ensure the JSON files are always written to database/data/ relative to the database module itself, regardless of which directory the server is started from. This fixed a subtle but critical bug where running npm run dev from the /backend directory was causing empty JSON files to be created in the wrong location.

 

4.3 REST API Endpoints

The backend exposes a clean RESTful API organised into three route groups:

Authentication Routes (/api/auth)

Method Endpoint Description Auth Required
POST /api/auth/register Create a new user account No
POST /api/auth/login Login and receive a JWT No
GET /api/auth/me Get current user profile Yes

Auction Routes (/api/auctions)

GET /api/auctions List all auctions (with populated user data) No
GET /api/auctions/:id Get a single auction by ID No
POST /api/auctions Create a new auction listing Yes

GET /api/auctions/won Get all auctions won by the current user Yes

Bid Routes (/api/bids)

Method Endpoint Description Auth Required
POST /api/bids Place a bid on an auction Yes
GET /api/bids/:auctionId Get all bids for an auction No

 

4.4 Authentication Middleware

Protected routes are guarded by the protect middleware in authMiddleware.js. The middleware extracts the JWT from the Authorization: Bearer <token> request header, verifies it against the JWT_SECRET environment variable using the jsonwebtoken library, and then fetches the corresponding user from the JSON database. If the token is missing, expired, or invalid, the middleware returns a 401 Unauthorized response immediately. If valid, it attaches the user object (with the password field stripped out) to req.user for downstream controller access.

 

4.5 Real-Time Communication Architecture

The real-time layer is built on Socket.IO 4, which establishes a WebSocket connection between the browser and the server, falling back to HTTP long-polling if WebSockets are unavailable.

The server manages two types of socket rooms:

  1. Auction Rooms: Named after the auction’s _id (e.g., "0e356b91f9ec7c82f495a8b7"). When a user opens an auction detail page, the frontend automatically emits join-auction with the auction ID, and the server adds that socket to the room. All new-bid and auction-ended events are broadcast to this room, so every viewer of that auction receives them simultaneously.

  2. Personal User Rooms: Named after the user’s _id. When a logged-in user’s browser connects, it also emits join-user with their ID. The server joins that socket to their personal room. This allows the endAuction controller to send the auction-won event directly and privately to just the winning bidder’s browser — like a whisper in a crowded room.

The Socket.IO server instance (io) is attached to every Express request via a middleware that does req.io = io. This allows any controller to emit events without importing the server module, keeping the architecture loosely coupled.

 

4.6 Bid Validation Logic

When a POST /api/bids request arrives, the placeBid controller performs several sequential validations before saving anything:

  1. Presence check: Both auctionId and amount must be provided.
  2. Auction existence: The referenced auction must exist in the database.
  3. Auction liveness: The auction’s status must not be 'ended' and its endTime must be in the future.
  4. Amount validity: The bid amount must be strictly greater than the current highest bid. A bid equal to or less than the current price is rejected.

Only after all four checks pass does the server create a Bid record, update the Auction record’s currentHighestBid and currentHighestBidder fields, and broadcast the new-bid socket event.

5.3 Sample Code — Placing a Bid (Frontend)

// frontend/src/components/NoteEditor.tsx

const placeBid = async () => {
  setError(''); setSuccess('');
  const num = parseFloat(amount);

  if (!num || num <= auction.currentHighestBid) {
    setError(`Enter an amount greater than ₹${auction.currentHighestBid.toLocaleString()}`);
    return;
  }

  setPlacing(true);
  try {
    await axios.post(`${API}/api/bids`, {
      auctionId: auction._id,
      amount: num,
    });
    setAmount('');
    setSuccess(`Bid of ₹${num.toLocaleString()} placed successfully!`);
    setTimeout(() => setSuccess(''), 4000);
  } catch (e) {
    const err = e as { response?: { data?: { message?: string } } };
    setError(err.response?.data?.message ?? 'Failed to place bid. Please try again.');
  } finally {
    setPlacing(false);
  }
};

 

5.4 Sample Code — Receiving Real-Time Bids (Socket.IO)

// frontend/src/components/NoteEditor.tsx

useEffect(() => {
  const socket = io(API, { auth: { token } });
  socketRef.current = socket;

  // Join the auction room and personal user room
  socket.emit('join-auction', initial._id);
  if (user?._id) socket.emit('join-user', user._id);

  // Handle incoming bids from other users
  socket.on('new-bid', (payload) => {
    setAuction((prev) => ({
      ...prev,
      currentHighestBid: payload.amount,
      currentHighestBidder: {
        _id: payload.bidder.id,
        name: payload.bidder.name
      },
    }));
    setBids((prev) => [
      {
        _id: payload.bidId,
        auction: initial._id,
        bidder: { _id: payload.bidder.id, name: payload.bidder.name },
        amount: payload.amount,
        createdAt: payload.timestamp,
      },
      ...prev,
    ]);
    setNewBidFlash(true);
    setTimeout(() => setNewBidFlash(false), 800);
  });

  // Handle winner notification — only fires for the winning bidder
  socket.on('auction-won', (payload) => {
    setWinnerBanner({
      auctionName: payload.productName,
      amount: payload.finalBid,
    });
    toast(`🏆 You won "${payload.productName}"!`, 'success');
  });

  return () => {
    socket.emit('leave-auction', initial._id);
    socket.disconnect();
  };
}, [initial._id, token, user?._id, toast]);

 

5.5 Sample Code — Ending an Auction (Backend Controller)

// backend/src/controllers/auctionController.js

export const endAuction = async (req, res) => {
  try {
    const auction = await Auction.findById(req.params.id);
    if (!auction) return res.status(404).json({ message: 'Not found' });

    // Only the creator can close their own auction
    if (auction.createdBy._id !== req.user._id && auction.createdBy !== req.user._id)
      return res.status(403).json({ message: 'Forbidden' });

    // Mark as ended and record the winner
    auction.status = 'ended';
    auction.winner = auction.currentHighestBidder;
    await Auction.save(auction);

    // Broadcast to all viewers of this auction
    req.io.to(req.params.id).emit('auction-ended', {
      auctionId: auction._id,
      winner: auction.winner,
      finalBid: auction.currentHighestBid,
    });

    // Private notification only to the winner
    if (auction.winner) {
      const winnerId = auction.winner._id || auction.winner;
      req.io.to(winnerId).emit('auction-won', {
        auctionId: auction._id,
        productName: auction.productName,
        finalBid: auction.currentHighestBid,
      });
    }

    res.json(auction);
  } catch (err) {
    res.status(500).json({ message: err.message });
  }
};

 

5.6 Sample Code — The JSON Database Layer

// database/jsonDb.js (excerpt)

export const Auction = {
  find: async () => {
    const users = db.getUsers();
    const now = new Date();
    return db.getAuctions().map(a => {
      const computedStatus = new Date(a.endTime) > now ? 'active' : 'ended';
      return {
        ...a,
        status: a.status || computedStatus,
        // Populate creator and bidder objects from users array
        createdBy: users.find(u => u._id === a.createdBy) || a.createdBy,
        currentHighestBidder:
          users.find(u => u._id === a.currentHighestBidder) || a.currentHighestBidder,
      };
    }).sort((a, b) => new Date(b.createdAt) - new Date(a.createdAt));
  },

  save: async (auctionObj) => {
    const auctions = db.getAuctions();
    const idx = auctions.findIndex(a => a._id === auctionObj._id);
    if (idx !== -1) {
      // Un-populate before saving — store only IDs, not full objects
      const toSave = { ...auctionObj };
      if (toSave.createdBy?._id) toSave.createdBy = toSave.createdBy._id;
      if (toSave.currentHighestBidder?._id)
        toSave.currentHighestBidder = toSave.currentHighestBidder._id;
      auctions[idx] = toSave;
      db.saveAuctions(auctions);
    }
  },
};

 

5.7 Winner Notification Banner

When Jabbar, who listed a phone for auction, decides that Alice’s bid is satisfactory and clicks “Close Auction & Accept Bid,” the following sequence of events happens in under 200 milliseconds:

  1. The browser calls PATCH /api/auctions/:id/end.
  2. The server validates that Jabbar is indeed the creator, then sets auction.status = 'ended' and auction.winner = alice._id.
  3. The server emits auction-ended to the auction room — Jabbar’s own screen updates, the bid panel shows “This auction has ended.”
  4. The server emits auction-won to Alice’s personal socket room — because Alice had previously joined this room when she connected.
  5. Alice’s browser receives the event and renders a full-screen modal: “Congratulations! You won the auction for iPhone 15 Pro. Final bid: ₹85,000.”
  6. A toast notification also appears in the corner of her screen, persisting for 3.5 seconds.

 

6. Conclusion

 

Building BidVerse was as much an exercise in systems thinking as it was in writing code. The challenge was not just to make a working auction platform — there are many tutorials that get you 80% of the way there — but to make one that felt real. Real auctions have tension. They have a moment when the hammer drops and somebody wins. They have that electric feeling of watching a number go up in real time.

Socket.IO was the key that unlocked this. Before adding real-time communication, the prototype was just a form-over-a-database application. After plugging in WebSockets, it transformed into something that felt alive. Watching the bid history update without touching the keyboard — seeing another user’s name appear at the top of the leaderboard — that changed everything about how the application felt to use.

The decision to use JSON files instead of MongoDB was initially a pragmatic one, made to avoid environment setup friction. But it turned out to be a valuable learning exercise in its own right. Implementing manual “populate” (joining user data to auction records) without an ORM’s help forced a deeper understanding of how relational data works at a fundamental level. The importance of storing only IDs in the JSON files and resolving them to full objects at query time — mirroring exactly what MongoDB’s populate() does behind the scenes — was a valuable insight.

The authentication system using JWT reinforced the stateless nature of HTTP and why tokens are superior to server-side sessions for APIs that need to scale. The protect middleware pattern — a single function that wraps any route and gates it behind authentication — is a clean, reusable solution that could be dropped into any Node.js project without modification.

Looking ahead, BidVerse has a clear path to production readiness. The JSON database can be replaced with MongoDB Atlas by swapping jsonDb.js with the Mongoose models that already exist in the codebase — a change that would require no modifications to the routes or controllers. The bcrypt + JWT authentication system is already production-grade. The deployment story is straightforward: the backend would be hosted on Render and the frontend on Vercel.

In summary, BidVerse successfully demonstrates the power of combining React’s component model with Socket.IO’s event-driven architecture to build a genuinely interactive, real-time web experience. It is not a toy — it is a foundation.
